# Dashboard

Read-only Web demo area.

The presentation demo should load completed artifacts from `baseline_runs/`,
`runs/`, and `case_studies/`. It must not trigger real crawling, Qwen inference,
or fake progress bars during the demo.

Expected first version:

- simple landing page
- completed channel examples
- per-channel dashboard tabs
- baseline percentile views
- cold report / owner-facing report views
- figures and tables generated by the analyzer

The real long-running pipeline remains in `scripts/` and should be documented
in `README.md`.

## Build Data

```bash
python3 scripts/build_dashboard_index.py
```

This writes the read-only JSON packets consumed by the Web UI:

```text
dashboard_data/index.json
dashboard_data/channels/<slug>.json
```

## Run Locally

```bash
python3 dashboard/server.py --host 127.0.0.1 --port 8765
```

Open:

```text
http://127.0.0.1:8765/
```

This is the default path after cloning the repository.

## Run On LAN

```bash
python3 dashboard/server.py --host 0.0.0.0 --port 8765
```

Then open from another device on the same network:

```text
http://<LAN_IP>:8765/
```

LAN addresses are machine-local. Do not hard-code them into frontend files or
shared dashboard data.

## External Access

Binding to `0.0.0.0` only makes the server listen on all local interfaces. It
does not by itself make the dashboard reachable from the public Internet.

For direct public-IP access, the router must forward a public TCP port to this
machine's local dashboard port and the local firewall must allow that port.
Those settings belong to the deployment machine, not to the GitHub repo.

For presentation use, a Cloudflare Tunnel is usually safer than exposing this
machine directly. The dashboard is read-only, but it still serves local project
artifacts.
